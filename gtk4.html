Because it took a couple of weeks for me to just correctly write and understand the hello, world program written in the official documentation. Anyway, we're going to fix the second problem in this chapter and complete the first version of hello, world program. Let me copy the hello-gtk.c code here from the previous chapter. 

#include<gtk/gtk.h>

int main(int argc, char *argv[]) {
  gtk_init(&argc, &argv);

  /**
   * 1. Create a variable that will hold a window.
   * 2. Create an actual window.
   * 3. Display the created window.
  **/
  GtkWidget *window;
  window = gtk_window_new(GTK_WINDOW_TOPLEVEL);
  gtk_window_set_title(GTK_WINDOW(window), "hello, world");
  gtk_widget_show(window);

  gtk_main();
  return 0;
}

Now, what we want with this program is, it should terminate the process (running in terminal) when clicked on close or x icon. In order to do that, we need to catch 'click on the close icon' signal or window's destroy signal.

First thing first, how I know that 'click on close icon' signal is the window's destroy signal? The answer is documentation and little experience with GTK. When you open the documentation for a specific widget for example window, you'll mostly see the section with title 'Signals' within the page and it lists the signals that particular widget emit. Don't assume that the mentioned signals are the only signals that the given widget emit. It also emits signals inherit from parent widget. For example, you might not see the destroy signal here but it is available from the widget.

Now you might have another question like, signal? What is it? GTK is an event-driven GUI toolkit where the flow of execution determines by events or can be modified by events. Do you know the event-driven programming? You might know if you're coming from JavaScript. If not, here is a simple description for you.

In event-driven programming, if you do something, an event will happen. If an event happens, it will emit the signal. You can catch this signal and write the code against it. For example, if you click on the button, a clicked event will happen and it emits the signal. Then you can catch this signal and write the code against it to do something interesting in terms of C functions. In this way, something will happen when you click on the button.

Back to our program, when we clicked on the close icon, a clicked event happens and it emits the destroy signal. We need to catch this signal and write the code (as a function) that run against this signal. To do so, g_signal_connect() function is used. This function takes four parameters and following is the signature of it: 

g_signal_connect(instance, detailed_signal, c_handler, data);

Before we go further and talk about the exact values that we're going to pass to this function, you might raise your voice on this function name because I previously told you that all functions name starts with gtk_ whereas this starts with g_. Let me tell you that what I told you about the naming convention on GTK functions is correct and what I wrote here as a function is also correct. But, the thing is, g_signal_connect() function is not from GTK it is from GLib.

Now, what is GLib? It is one of the low-level C utilities library and GTK heavily depends on it. When you include gtk/gtk.h in your program, it automatically includes this GLib or glib.h library in your program via this gtk.h header file. So, you can directly use the functions from GLib (things do not end here, GLib has GObject which is also included here and from this GObject this g_signal_connect() is coming. But, let's leave this complicated dependency discussion for future chapters when you have enough background).

In the future, I'll talk about GLib in detail. But for now, let's go with this - the signal mechanism is part of GLib and due to this we're going to use GLib's functions such as g_singal_connect(). Also, if the function is from GLib, it starts with g_ and then the name of component or mechanism such as signal followed by verb or action that the function does such as connect.

Now, after this background information, let's talk about the exact values we need to pass to the g_singal_connect() function. First, it takes instance or widget, which emits the signal and in our case, it is a window. 

g_signal_connect(window, detailed_signal, c_handler, data);

But, we need to cast this variable into the actual window and we know how to do it, right? Using GTK_WINDOW macro. 

g_signal_connect(G_WINDOW(window), "destroy", c_handler, data);

We're interested in catching the signal emitted by the window. But which signal? It is destroy. And don't worry GTK already written code for this specific scenario - if you click on the close icon, it'll emit signal. Let's write this name of the signal as string e.g. "destroy". 

g_signal_connect(G_WINDOW(window), "destroy", c_handler, data);

Good! Now c_handler means a name of the function that will be called when window emits destroy signal. Again, you just have to write the name of the function, not a function call. In other words, if you have a function with name app_quit() then you have to write app_quit not app_quit() here (in other words, function name without parenthesis). Also, this g_singal_connect() accepts only GCallback types of function and due to this, you have to cast the function using G_CALLBACK macro.

Don't forget that that function name we're going to pass here is the normal C function which we have to write and it'll run against the window's destroy signal. Here when we click on a close icon, it should terminate or quit the application. Based on this requirement, I think the function with name app_quit() is suitable to understand its purpose. So, let's pass this name and cast it using the G_CALLBACK macro. 

g_signal_connect(G_WINDOW(window), "destroy", G_CALLBACK(app_quit), data);

Finally, you can pass some data to this app_quit() function when called. If you don't want to pass, you can just write NULL because, in GTK, parameters are not optional, you must have to pass something. In our case, let's go with NULL for now. 

g_signal_connect(G_WINDOW(window), "destroy", G_CALLBACK(app_quit), NULL);

And this is how we're going to connect the specific widget's signal to our custom code via a callback function. This callback word is from event-driven programming, meaning a function that will be called when an event happens. Let's copy the about function call in our program and we should put this after the creation of window as follows: 

#include<gtk/gtk.h>

int main(int argc, char *argv[]) {
  gtk_init(&argc, &argv);

  /**
   * 1. Create a variable that will hold a window.
   * 2. Create an actual window.
   * 3. Display the created window.
  **/
  GtkWidget *window;
  window = gtk_window_new(GTK_WINDOW_TOPLEVEL);
  gtk_window_set_title(GTK_WINDOW(window), "hello, world");
  g_signal_connect(GTK_WINDOW(window), "destroy", G_CALLBACK(app_quit), NULL);

  gtk_widget_show(window);

  gtk_main();
  return 0;
}

Does our program is now ready to run? No! Because we haven't written the app_quit() function. We can write this function after or before the main() function. If we write the app_quit() function after main() (or more specifically after the call), we have to first declare the function using its signature at the top. But, if we write the function before main(), then we can write the function directly. This is related to C, not GTK and I assume that you already know. What I just did here is revised your C knowledge! I'm going to write the function before its call. 

#include<gtk/gtk.h>

app_quit() {

}

int main(int argc, char *argv[]) {
  gtk_init(&argc, &argv);

  /**
   * 1. Create a variable that will hold a window.
   * 2. Create an actual window.
   * 3. Display the created window.
  **/
  GtkWidget *window;
  window = gtk_window_new(GTK_WINDOW_TOPLEVEL);
  gtk_window_set_title(GTK_WINDOW(window), "hello, world");
  g_signal_connect(GTK_WINDOW(window), "destroy", G_CALLBACK(app_quit), NULL);

  gtk_widget_show(window);

  gtk_main();
  return 0;
}

I mentioned earlier that this function is not the normal function but a special type of function that will run when destroy signal caught and due to this, you need to define this function in the way destroy signal points out and this is how destroy signal wants its function! 

void user_function(GtkWidget *object, gpointer user_data)

First thing first, let's update the function name with app_quit. 

void app_quit(GtkWidget *object, gpointer user_data)

This function accept the object which emit the signal and we normally call it widget so I'll use widget name instead of object. 

void app_quit(GtkWidget *widget, gpointer user_data)

This widget is a reference to the window and you can do all sorts of stuff with a window within a function (and we surely do in future chapters). Finally, this gpointer is a pointer to data we passed from g_singal_connect() function as the last argument. We can also call it just data if we want and I'll do. 

void app_quit(GtkWidget *widget, gpointer data)

So, this is the signature I'll going to use for this and future programs for the window's destroy signal. Also, remember that not all callback functions are written in this way, it is based on signal and I'll talk about this in future chapters. Let's update the function signature with the mentioned one. 

#include<gtk/gtk.h>

void app_quit(GtkWidget *widget, gpointer data) {

}

int main(int argc, char *argv[]) {
  gtk_init(&argc, &argv);

  /**
   * 1. Create a variable that will hold a window.
   * 2. Create an actual window.
   * 3. Display the created window.
  **/
  GtkWidget *window;
  window = gtk_window_new(GTK_WINDOW_TOPLEVEL);
  gtk_window_set_title(GTK_WINDOW(window), "hello, world");
  g_signal_connect(GTK_WINDOW(window), "destroy", G_CALLBACK(app_quit), NULL);

  gtk_widget_show(window);

  gtk_main();
  return 0;
}

Now, inside this function, we can write the code as whatever we want or the code that quit the application in our case. Actually, the problem is with the gtk_main() loop because it is running in the background and we need to terminate it when clicked on close icon. This will automatically close the process in a terminal and this can be done using gtk_main_quit() function from GTK. So, what we have to do is, just call this function from our app_quit() function as follow: 

#include<gtk/gtk.h>

void app_quit(GtkWidget *widget, gpointer data) {
  gtk_main_quit();
}

int main(int argc, char *argv[]) {
  gtk_init(&argc, &argv);

  /**
   * 1. Create a variable that will hold a window.
   * 2. Create an actual window.
   * 3. Display the created window.
  **/
  GtkWidget *window;
  window = gtk_window_new(GTK_WINDOW_TOPLEVEL);
  gtk_window_set_title(GTK_WINDOW(window), "hello, world");
  g_signal_connect(GTK_WINDOW(window), "destroy", G_CALLBACK(app_quit), NULL);

  gtk_widget_show(window);

  gtk_main();
  return 0;
}

That's it, this function will take care of remaining things - close the application as well as the process. That's it! Let's run the program: 

$ gcc `pkg-config --cflags --libs gtk+-3.0` hello-gtk.c -o hello-gtk
$ ./hello-gtk

Now, click on close button from the window and it should terminate the process in terminal. For me its works, how about you?

In case if you want, you can also print the message in console before terminating the process like - "terminating the application..." using g_print() function as follows: 

#include<gtk/gtk.h>

void app_quit(GtkWidget *widget, gpointer data) {
  g_print("terminating the application...");
  gtk_main_quit();
}

int main(int argc, char *argv[]) {
  gtk_init(&argc, &argv);

  /**
   * 1. Create a variable that will hold a window.
   * 2. Create an actual window.
   * 3. Display the created window.
  **/
  GtkWidget *window;
  window = gtk_window_new(GTK_WINDOW_TOPLEVEL);
  gtk_window_set_title(GTK_WINDOW(window), "hello, world");
  g_signal_connect(GTK_WINDOW(window), "destroy", G_CALLBACK(app_quit), NULL);

  gtk_widget_show(window);

  gtk_main();
  return 0;
}

Now, if you run this program and then close it, you'll also see the message in terminal as follows: 

$ gcc `pkg-config --cflags --libs gtk+-3.0` hello-gtk.c -o hello-gtk
$ ./hello-gtk
terminating the application...
